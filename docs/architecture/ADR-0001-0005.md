## ADR-0001: 选用 Hermes Agent 作为中台底座

### 元信息

| 字段 | 值 |
|:----|:----|
| **ID** | ADR-0001 |
| **标题** | 选用 Hermes Agent 作为中台底座 |
| **状态** | ✅ Accepted |
| **决策日期** | 2026-07-18 |
| **最后更新** | 2026-07-18 |
| **影响范围** | 架构选型、基础设施、开发工具链 |
| **发起人** | 架构组 |

### Context (背景)

全讯AI中台需要承载 10 个专业化 AI Agent，每个 Agent 有不同的身份、权限、模型和工具配置。核心需求：

- **多 Agent 管理**: 支持 10+ 独立 Agent 的运行和生命周期管理
- **Profile 隔离**: 每个 Agent 需要独立的配置（模型、工具、Prompt、身份凭证）
- **MCP 协议兼容**: 工具层需要标准化接入，支持未来可插拔扩展
- **弹性扩展**: 未来可能增加 Agent 数量，需要低成本扩容
- **轻量部署**: 全部运行在一台 ThinkPad X1 Extreme 笔记本上，资源有限

### Decision (决策)

**选用 Hermes Agent v0.18.2 作为全讯AI中台的核心底座框架。**

选择理由与评估结论：

| 维度 | Hermes Agent | Dify | Coze | OpenClaw |
|:----|:------------|:----|:----|:--------|
| **开源** | ✅ MIT 协议完全开源 | ✅ 开源 | ❌ 闭源 SaaS | ✅ 开源 |
| **多 Profile** | ✅ 原生支持 | ❌ 需 Hack | ❌ 不支持 | ❌ 不支持 |
| **MCP 协议** | ✅ 原生支持 | ⚠️ 社区插件 | ❌ 不支持 | ❌ 不支持 |
| **本地部署** | ✅ 极轻量 | ⚠️ 重量级 | ❌ 强制云端 | ✅ 本地 |
| **资源占用** | ~50MB/Agent | ~2GB+ | N/A | ~500MB+ |
| **工具扩展** | ✅ MCP Server 即插即用 | ⚠️ 插件市场 | ✅ 受限 | ⚠️ 自定义 |
| **社区活跃** | ✅ Nous Research 维护 | ✅ 活跃 | N/A | ⚠️ 一般 |

### Consequences (结果)

**正面**:
- 10 个独立 Profile 开箱即用，每个 Profile 独立配置模型、MCP、Prompt
- MCP 协议原生支持，11 个 MCP Server 无缝集成
- 极低资源消耗（单 Agent ~50MB 内存），适合笔记本部署
- MIT 开源协议，无商业限制

**负面**:
- 文档相对较新，部分高级特性需要有源码阅读能力
- 社区规模比 Dify 小，生态工具少
- 缺少可视化编排界面（纯配置驱动）

### Compliance (合规检查)

| 检查项 | 状态 | 备注 |
|:------|:----:|:-----|
| 开源合规 | ✅ | MIT 协议，无使用限制 |
| 数据安全 | ✅ | 全部本地运行，数据不出域 |
| 许可合规 | ✅ | 企业级使用无限制 |
| 依赖风险 | ⚠️ | Python 生态，需维护 venv |

## ADR-0002: 多 Profile 独立 Gateway 架构

### 元信息

| 字段 | 值 |
|:----|:----|
| **ID** | ADR-0002 |
| **标题** | 多 Profile 独立 Gateway 架构 |
| **状态** | ✅ Accepted |
| **决策日期** | 2026-07-18 |
| **最后更新** | 2026-07-18 |
| **影响范围** | 部署架构、运维、消息路由 |
| **发起人** | 架构组 |

### Context (背景)

全讯AI中台有 10 个独立 Agent，每个 Agent 对应一个飞书 App（独立身份、独立回调地址）。架构上需要解决：

- **身份隔离**: 飞书 App A 的消息不能错误路由到 App B
- **权限隔离**: 不同 Agent 有不同的 MCP 权限（如财务 Agent 才能访问 Tushare）
- **故障隔离**: 一个 Agent 崩溃不应影响其他 Agent
- **独立配置**: 每个 Agent 使用不同的模型、System Prompt、MCP Server
- **独立扩展**: 未来可按需为高负载 Agent 分配更多资源

### Decision (决策)

**采用 10 个独立 Profile，每个 Profile 运行一个独立的 Gateway 进程，分别绑定端口 13000-13010。**

```
端口分配方案:
  Profiles:
    - default          → 13000
    - caiwuzhushou     → 13001
    - zhuli            → 13002
    - stockexpert      → 13003
    - manager          → 13004
    - productexpert    → 13005
    - engineeringdir   → 13006
    - marketingdir     → 13007
    - ecommerceops     → 13008
    - operationsdir    → 13009
```

备选方案比较：

| 方案 | 隔离性 | 扩展性 | 资源消耗 | 复杂度 |
|:----|:-----:|:------:|:--------:|:-----:|
| ✅ **独立 Gateway** | ★★★★★ | ★★★★★ | ★★★★☆ | ★★☆☆☆ |
| 单 Gateway 多 Session | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★★★ |
| 共享 Gateway + 名字空间 | ★★★☆☆ | ★★★★☆ | ★★★★☆ | ★★★☆☆ |

### Consequences (结果)

**正面**:
- 完全故障隔离：某个 Gateway 进程 OOM 不影响其他 9 个 Agent
- 独立配置：每个 Profile 的 `.env`、`hermes.yaml`、MCP 配置完全独立
- 独立扩展：高频 Agent（如 stockexpert）可单独分配更多资源
- 独立版本：不同 Profile 可以使用不同版本的 MCP Server 配置

**负面**:
- 资源开销：10 个 Gateway 进程约 500MB 内存（50MB × 10）
- 运维复杂度：需要管理 10 个 systemd service 或 supervisor 进程
- 端口管理：需要预留 13000-13009 端口范围

### Compliance (合规检查)

| 检查项 | 状态 | 备注 |
|:------|:----:|:-----|
| 进程隔离 | ✅ | 独立进程，操作系统级隔离 |
| 配置隔离 | ✅ | 每个 Profile 独立配置文件 |
| 端口冲突 | ✅ | 明确规划端口范围 |
| 监控覆盖 | ⚠️ | 需补充进程健康检查机制 |

## ADR-0003: Qdrant 本地知识库 + 两级审核机制

### 元信息

| 字段 | 值 |
|:----|:----|
| **ID** | ADR-0003 |
| **标题** | Qdrant 本地知识库 + 两级审核机制 |
| **状态** | ✅ Accepted |
| **决策日期** | 2026-07-18 |
| **最后更新** | 2026-07-18 |
| **影响范围** | 知识管理、数据安全、检索架构 |
| **发起人** | 架构组 |

### Context (背景)

企业知识库需要管理 10 万 + 条结构化知识条目，核心约束：

- **数据安全**: 企业知识不得离开本地网络
- **知识质量**: 避免垃圾知识污染检索结果
- **向量维度**: 嵌入模型 qwen3-embedding:4b 输出维度 2560
- **性能要求**: 10 万条数据下，检索延迟 < 200ms
- **增量更新**: 需要支持持续写入新知识

### Decision (决策)

**选用 Qdrant 本地部署（Windows 本机 172.29.192.1:6333）+ candidate/published 两级状态管理。**

备选方案比较：

| 维度 | ✅ Qdrant | Milvus | Weaviate | Elasticsearch |
|:----|:---------|:-------|:---------|:-------------|
| **部署复杂度** | ★☆☆☆☆ 极简 | ★★★★☆ 复杂 | ★★★☆☆ 中等 | ★★★★☆ 复杂 |
| **资源占用** | ~200MB | ~2GB+ | ~500MB | ~2GB+ |
| **2560 维支持** | ✅ 原生 | ✅ 原生 | ✅ 原生 | ✅ 需插件 |
| **增量更新** | ✅ 高效 | ✅ 高效 | ✅ 高效 | ✅ 一般 |
| **本地部署** | ✅ 纯本地 | ✅ 可本地 | ✅ 可本地 | ✅ 可本地 |
| **Docker 依赖** | ⚠️ Windows 版无需 | 需要 | 需要 | 需要 |
| **查询延迟 (10万)** | ~50ms | ~30ms | ~60ms | ~100ms |

### Consequences (结果)

**正面**:
- Windows 本机部署，数据不出域，完全满足企业安全要求
- 两级审核（candidate → published）防止知识污染
- qwen3-embedding:4b（dim=2560）矢量兼容性好
- 10 万条数据下实际检索延迟 50-200ms，满足实时性要求
- 增量写入零中断，持续同步飞书知识库

**负面**:
- Qdrant 部署在 Windows 侧，WSL 通过 host 网络访问（172.29.192.1:6333）
- 两级审核需额外开发（知识入库工作流）
- Qdrant 的全文检索能力弱于 Elasticsearch（但向量检索是其强项）
- 单节点部署无高可用（笔记本关机即不可用）

### Compliance (合规检查)

| 检查项 | 状态 | 备注 |
|:------|:----:|:-----|
| 数据不出域 | ✅ | 纯本地部署，无外部网络调用 |
| 知识审核 | ✅ | candidate → published 两级 |
| 隐私保护 | ✅ | 无用户数据外泄风险 |
| 备份策略 | ⚠️ | 需补充 Qdrant snapshot 备份 |

## ADR-0004: 多模型 Provider 策略 (DeepSeek+GLM 双轨)

### 元信息

| 字段 | 值 |
|:----|:----|
| **ID** | ADR-0004 |
| **标题** | 多模型 Provider 策略 (DeepSeek+GLM 双轨) |
| **状态** | ✅ Accepted |
| **决策日期** | 2026-07-18 |
| **最后更新** | 2026-07-18 |
| **影响范围** | Agent 配置、成本模型、推理质量 |
| **发起人** | 架构组 |

### Context (背景)

10 个 Agent 承担不同类型的任务，对模型能力的需求差异很大：

| 场景 | 需求 | 日预估调用量 |
|:----|:----|:-----------:|
| 通用问答（default） | 快速、低成本 | 200+ |
| 财务分析（caiwuzhushou） | 中文金融领域、中等推理 | 50+ |
| 投资研究（stockexpert） | 强推理、长上下文 | 20+ |
| 代码审查（engineeringdir） | 代码理解、结构化 | 30+ |
| 营销创作（marketingdir） | 中文文案、创意生成 | 30+ |

核心约束：
- **成本**: 总 API 费用控制在 $50-100/月
- **质量**: 复杂任务需要顶级推理能力
- **延迟**: 日常问答要求 < 5s 响应
- **中文**: 部分场景需要中文优化模型

### Decision (决策)

**采用 DeepSeek Flash 日常 + DeepSeek Pro 复杂 + GLM-5.2 中文的三轨模型策略，搭配 qwen3-embedding:4b 本地嵌入。**

| Agent | 模型 | 月估算 Token | 月成本 |
|:------|:----:|:-----------:|:------:|
| 💬 default | DS Flash | ~5M | ~$10 |
| 💰 caiwuzhushou | DS Flash | ~3M | ~$6 |
| 🤖 zhuli | DS Flash | ~2M | ~$4 |
| 📈 stockexpert | DS Pro | ~2M | ~$20 |
| 🧭 manager | DS Pro | ~1M | ~$10 |
| 🏗️ productexpert | DS Flash | ~2M | ~$4 |
| 🛠️ engineeringdir | GLM-5.2 | ~2M | ~$8 |
| 📣 marketingdir | GLM-5.2 | ~2M | ~$8 |
| 🛒 ecommerceops | DS Flash | ~2M | ~$4 |
| ⚙️ operationsdir | DS Flash | ~2M | ~$4 |
| **总计** | | **~23M** | **~$78** |

备选方案比较：

| 方案 | 月成本 | 质量 | 维护复杂度 |
|:----|:-----:|:----:|:---------:|
| ✅ **多模型策略** | $50-100 | ★★★★★ | ★★★☆☆ |
| 单一模型（全部 DS Pro） | $300-500 | ★★★★☆ | ★☆☆☆☆ |
| 自建模型（本地推理） | $0 | ★★☆☆☆ | ★★★★★ |
| 全部 GLM | $80-150 | ★★★☆☆ | ★★☆☆☆ |

### Consequences (结果)

**正面**:
- 成本最优化：相比全部使用 DS Pro，节省 80% API 费用
- 能力最匹配：复杂任务用 Pro，日常用 Flash，中文用 GLM
- 本地嵌入零成本：qwen3-embedding 免费运行，无外部依赖
- 故障容错：一个 Provider 不可用时，可切换其他 Provider

**负面**:
- 多 Provider 密钥管理：需要维护 3 套 API Key + 计费
- 模型差异：不同 Agent 使用不同模型，回复风格不一致
- GLM-5.2 的输出质量控制需要额外 Prompt 工程
- 依赖外部 API 可用性（DeepSeek / GLM 服务稳定性）

### Compliance (合规检查)

| 检查项 | 状态 | 备注 |
|:------|:----:|:-----|
| 成本合规 | ✅ | 预算内 $50-100/月 |
| 数据隐私 | ⚠️ | 部分 Agent 数据经 DeepSeek/GLM API（通过 HTTPS）|
| Provider 替换 | ✅ | 可在 Profile 中一键切换模型 |
| API Key 安全 | ⚠️ | 需使用环境变量管理，避免 Hardcode |

## ADR-0005: 通过 MCP 协议统一工具接入

### 元信息

| 字段 | 值 |
|:----|:----|
| **ID** | ADR-0005 |
| **标题** | 通过 MCP 协议统一工具接入 |
| **状态** | ✅ Accepted |
| **决策日期** | 2026-07-18 |
| **最后更新** | 2026-07-18 |
| **影响范围** | 工具链、Agent 开发、第三方集成 |
| **发起人** | 架构组 |

### Context (背景)

全讯AI中台的 Agent 需要接入多样化的工具能力：

- **联网搜索**: AnySearch
- **网页爬虫**: Scrapling
- **知识检索**: Qdrant
- **办公协作**: 飞书
- **代码管理**: GitHub
- **金融数据**: Tushare
- **创意工具**: OpenMontage, Draw.io, Agent-Reach

核心需求：
- **标准化接入**: 所有工具通过统一的协议注册和使用
- **可插拔**: 添加/移除工具无需修改 Agent 代码
- **工具复用**: 一个工具可被多个 Agent 共享
- **生态兼容**: 未来可能接入更多第三方工具
- **权限控制**: 不同 Agent 可配置不同的工具集合

### Decision (决策)

**全部工具通过 MCP (Model Context Protocol) 协议统一接入，不直接调用底层 API。**

```
                          ┌──────────────────────────┐
                          │      Agent (任何Profile)   │
                          │  ┌────────────────────┐   │
                          │  │  MCP Client        │   │
                          │  │  (Hermes 内置)     │   │
                          │  └─────────┬──────────┘   │
                          └────────────┼──────────────┘
                                       │
                          ┌────────────┼──────────────┐
                          │  MCP 协议层               │
                          │  JSON-RPC 2.0             │
                          └────────────┼──────────────┘
                                       │
          ┌────────────────────────────┼──────────────────────────────┐
          │           ┌────────────────┼────────────────┐            │
          │           ▼                ▼                ▼            │
          │   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐    │
          │   │ AnySearch    │ │ Scrapling    │ │ Qdrant       │    │
          │   │ MCP Server   │ │ MCP Server   │ │ MCP Server   │    │
          │   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘    │
          │          │                │                │            │
          │   ┌──────┴───────┐ ┌──────┴───────┐ ┌──────┴───────┐    │
          │   │ Feishu       │ │ GitHub       │ │ Tushare      │    │
          │   │ MCP Server   │ │ MCP Server   │ │ MCP Server   │    │
          │   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘    │
          │          │                │                │            │
          │   ┌──────┴───────┐ ┌──────┴───────┐                       │
          │   │ 未来: 更多    │ │ 底层 API     │                       │
          │   │ MCP Server   │ │ HTTP/gRPC    │                       │
          │   └──────────────┘ └──────────────┘                       │
          └────────────────────────────────────────────────────────────┘
```

每个 MCP Server 封装一种能力，通过 JSON-RPC 2.0 与 Hermes Agent 通信：

```yaml
# MCP Server 配置示例 (hermes.yaml)
mcps:
  anysearch:
    type: stdio
    command: anysearch-mcp
  scrapling:
    type: stdio
    command: scrapling-mcp
  qdrant:
    type: stdio
    command: qdrant-mcp
  feishu:
    type: stdio
    command: feishu-mcp
```

备选方案比较：

| 维度 | ✅ MCP 协议 | 直接 API 调用 | Plugin 机制 |
|:----|:-----------|:-------------|:-----------|
| **标准化** | ★★★★★ 统一协议 | ★★☆☆☆ 各自为政 | ★★★☆☆ 半标准化 |
| **可插拔** | ★★★★★ 真即插即用 | ★☆☆☆☆ 需改代码 | ★★★★☆ 需注册 |
| **复用性** | ★★★★★ 跨 Agent 共享 | ★★☆☆☆ 重复实现 | ★★★☆☆ 框架绑定 |
| **生态** | ★★★★★ 社区增长中 | ★★★☆☆ 成熟但分散 | ★★☆☆☆ 框架绑定 |
| **学习成本** | ★★★☆☆ 中等 | ★☆☆☆☆ 简单 | ★★★☆☆ 中等 |
| **调试难度** | ★★★★☆ 统一日志 | ★★★☆☆ 各工具独立 | ★★★☆☆ 中等 |

### Consequences (结果)

**正面**:
- 工具即插即用：添加新的 MCP Server 只需在配置文件中注册
- 跨 Agent 共享：一个 MCP Server 可同时服务多个 Agent
- 标准化接口：所有工具通过统一的 tool list / tool call 接口
- 生态兼容：MCP 是标准化协议，未来可接入任何 MCP 兼容 Server
- 版本独立：MCP Server 可独立升级，不影响 Agent 运行

**负面**:
- 额外抽象层：MCP 协议带来轻微的性能开销（JSON-RPC 序列化/反序列化）
- 调试复杂度：工具调用链路变长（Agent → MCP Client → MCP Server → 底层 API）
- 工具清单同步：新工具需手动更新 Agent 的 System Prompt（告知 Agent 可用工具）
- MCP 生态仍在发展中，部分工具需要自行封装

### Compliance (合规检查)

| 检查项 | 状态 | 备注 |
|:------|:----:|:-----|
| 协议标准 | ✅ | 遵循 MCP Model Context Protocol 标准 |
| 接口统一 | ✅ | 全部通过 Tool Call 接口 |
